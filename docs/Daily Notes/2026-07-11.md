## Session — pool-asset save bug + default Folder template

- **Bug (owner report):** editing a Team asset threw *"Pool assets are workspace-visible — use hidden_from instead of shares."* Root cause: `POST /assets` returned the raw record with **no `_owner`/`_access` annotation**; the create→edit flip (`setAsset(created)`) then treated a fresh pool asset as personal — share selector rendered, Save sent `shared_with: []` (not None) → backend 400. Household had the same latent bug (owner's household edits were on re-opened, annotated list items — that's why it "worked"). Same class: a child created under an edit-shared foreign asset lost `_owner` → Save would call updateAccess → 403.
- **Fix (backend, router-level):** create response is now annotated like list/find — pool store → `_owner: team|household`, foreign store → `_owner: <owner name>`, both `_access: "edit"`; own store unannotated. This also fixes the flipped modal's other gating (Convert/Delete/Leave/parent-picker) for fresh pool/foreign assets. No frontend change needed.
