- **Content authored & shipped** in one source, **not** user-editable Brain markdown.

### Architecture — one source, three consumers
`app/backend/content/help.json` (`{sections, faq, support, whats_new}`) → `services/help_service.py`
(`get_content` / `as_text` w/ `/help#<id>` anchors / `capabilities_index` / onboarding get-set) →
(1) `routers/help.py` `GET /help/content|whats-new`, `GET/PUT /help/onboarding` → `pages/Help.jsx`;
(2) module-page ⓘ `components/HelpButton.jsx` deep-links; (3) AI `get_help` tool (in
`_READ_TOOLS`/`_RESEARCH_TOOLS`) + capability index injected in `chat._build_context`.

### Design notes / gotchas
- Onboarding read/write logic lives in `help_service` (router is thin) so tests don't import the auth
  chain (CI stubs `auth_service`).
- What's-New: fresh installs set a **silent baseline** (no "you updated" spam); announce only when the
  running version > last announced AND a matching `whats_new` entry exists; idempotent per version.
- Help is a hardcoded footer nav entry like Settings — **not** a module (no `require_module`, no
  `ALL_MODULES` entry, unguarded route), so every user (incl. guests) sees it.

### NOT deployed

What's-New broadcast fires at boot: since `_system/whats_new_state.json` won't exist yet, the first
relaunch sets a **silent baseline** (no broadcast). To actually test the broadcast, bump `VERSION` +
add a matching `whats_new` entry, then relaunch.
