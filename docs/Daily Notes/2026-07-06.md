## Session — chat archive truncation fix (bug 2 of 2)

- Root cause was **not** the step-trace assembly (original suspicion) — the backend saves full content. The archive **parse** was lossy: `parseSavedChat()` in `Chat.jsx` kept only lines starting with `**You**:`/`**AI**:` and dropped all continuation lines of multi-line messages. Continuing a saved chat loaded the truncated parse, and the 1.5 s auto-save then **overwrote the same archive** with it — permanent loss, lead-in line kept. Parser now accumulates continuation lines into the current message.
- Saved-chat viewer drawer had the same per-line rendering (`return null` on continuation lines) — now renders bubbles from `parseSavedChat()` with `whitespace-pre-wrap`.
- Adjacent cause: `HistoryMessage.content` cap of 5000 chars made auto-save 422 silently for long agent/research responses (catch is deliberately silent). Raised to 30,000 across `/chat`, `/chat/save`, `/chat/save-memory` so long responses save AND continued chats containing them pass validation. User input stays capped at 5000 via `ChatRequest.message`.
- Verified: frontend builds; Node round-trip script (serialize → parse → serialize → parse, multi-line + multi-paragraph messages) all-preserved; backend suite 202 passed; black/isort clean.
