## How the Update Flow Works

1. Admin visits Admin → Updates, sees current version vs latest GitHub release.
2. Clicks "Apply Update" → POST `/update/apply` writes `brain/_system/pending_update`.
3. Host-side `update.sh --watch` (or `--cron` via crontab) sees the flag.
4. Script: backup Brain → git pull → npm build → docker compose up --build → health check.
5. Health check passes → writes new version to `brain/_system/installed_version.json`.
6. Health check fails → git reset --hard + rebuild from prev commit (rollback).
7. Admin UI polls every 8s during pending/running state; shows log on demand.

**Auto-update setup:** `bash launch.sh --auto-update` installs the crontab entry. Or run `bash docker/update.sh --watch` manually in a persistent session.

## Decisions Made

- **Flag-file approach** chosen over Docker socket exec: more secure, no host filesystem mount required inside the container, consistent with the Brain-as-filesystem philosophy.
- **Git reset rollback** (not image snapshot): simpler, reliable, ~2 min to rebuild from previous source. Fast enough for a self-hosted app.
- **GITHUB_REPO** hardcoded as `logcoretech/logcoreOS` in update_service.py — set to `""` to disable checks (for forks or air-gapped installs).
- **Cache TTL 4 hours** — balances freshness vs GitHub API rate limits (60 unauthenticated req/hr).
- **Daily scheduler job** at noon just refreshes the cache. Push notification for admins deferred — the Admin UI badge is sufficient for now.
