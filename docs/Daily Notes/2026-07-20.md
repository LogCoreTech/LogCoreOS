## Separate session — mobile footer bottom-gap fix

Owner reported the mobile bottom footer nav had "doubled" padding — a photo showed
the icon row on the nav's normal dark background, then a visibly *lighter* gray
band below it before the true screen bottom (home-indicator area).

- Initial suspicion from the TASKS.md note (a doubled `env(safe-area-inset-bottom)`
  reserve) didn't hold up: grep found only one such declaration anywhere in the
  footer's render tree, and reproducing the exact `Layout.jsx` structure in
  Chromium with `Emulation.setSafeAreaInsetsOverride` (CDP) showed it being applied
  correctly once, nav flush to the bottom.
- **First fix attempt was wrong and had to be reverted**: swapped the `Layout.jsx`
  root from `h-[100dvh]` to `h-full` on the theory that `dvh` was the problem, but
  didn't verify against a live render first — `h-full` needs a non-`auto` height on
  every ancestor, and `#root` (the React mount div in `index.html`) has none, so
  the flex column collapsed to content height instead of filling the screen. Caught
  this by actually spinning up the app (backend venv + `uvicorn`, `npm run dev`,
  registered a test user through Setup, Playwright screenshot) rather than trusting
  the diff — the screenshot showed an even bigger blank gap than before. Reverted,
  committed, pushed immediately.
- The owner's own follow-up hypothesis nailed the real cause: **the body background
  expands to fill the screen correctly (`height: 100%`, unaffected by dynamic-viewport
  units); the footer's flex column doesn't reach as far**, because iOS/WebKit has
  known inconsistencies computing `dvh`'s bottom bound against the safe-area inset
  in standalone/PWA mode — `body`'s lighter color was showing through the shortfall
  below the nav's own (correctly single-reserved) darker background.
- Real fix: `#root { height: 100%; }` added to `index.css` so the plain percentage
  chain (`html`→`body`→`#root`) resolves properly, then `Layout.jsx` root switched
  to `h-full` (not `dvh` at all — sidesteps the dynamic-viewport/safe-area
  interaction entirely). Verified via the same live-app + Playwright/CDP setup, both
  light and dark mode: nav flush to the simulated safe-area bottom, zero gap,
  matching computed background colors confirmed via `getComputedStyle` (no
  `body`-color exposure in any run). Lint clean, `vite build` clean.
- **Not verified on real iOS hardware** — the sandbox has no notched device or
  actual WebKit engine, only Chromium with a CDP-simulated safe-area inset, which
  does not reproduce the underlying iOS-specific `dvh` quirk (Chromium's `dvh` is
  spec-correct and never showed the bug even before the fix). Owner confirms on
  next release.
- Merged straight to `master` at owner's explicit request (fast-forward from
  `claude/doubled-footer-mobile-qj4wys`, no conflicts) rather than going through a
  PR.
