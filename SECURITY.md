# Security Policy

## Reporting a Vulnerability

If you discover a security vulnerability in LogCoreOS, please **do not open a public GitHub issue**.

Report it privately by emailing: **support@logcoretech.com**

Include:
- A description of the vulnerability and its potential impact
- Steps to reproduce (proof of concept if possible)
- Affected versions or components

We aim to acknowledge reports within **48 hours** and provide a fix or mitigation within **7 days** for critical issues.

If you'd like credit, we'll name you in the `CHANGELOG.md` entry for the fix — just say so (and how you'd like to be credited) in your report. Anonymous reports are just as welcome.

---

## Scope

The following are in scope:

- **Authentication** — JWT handling, JTI revocation, bcrypt password storage
- **Authorization** — admin endpoint access, module access control, path traversal in Brain file reads/writes
- **Session security** — cookie flags (`Secure`, `HttpOnly`, `SameSite`), token expiry
- **Injection** — prompt injection via Brain file content injected into AI context, command injection in Bash hooks
- **CORS and headers** — misconfigured CORS origins, missing security headers
- **Self-hosting deployment** — `docker/.env` exposure, `auth.json` exposure, `SECRET_KEY` weaknesses

Out of scope:
- Vulnerabilities that require physical access to the server
- Denial-of-service attacks (rate limiting is enforced but DDoS is out of scope)
- Issues in third-party dependencies that have already been disclosed upstream (report those to the dependency maintainers)

---

## Security Design Notes

For developers and security researchers:

- Brain file content injected into AI prompts is wrapped in `<brain_data>` XML tags (see `routers/chat.py:_safe()`) to limit prompt injection surface
- User names are validated against `_NAME_RE` at creation time to prevent path traversal
- `trust_proxy_headers` defaults to `False` — only enable behind a verified reverse proxy
- `SECRET_KEY` is auto-generated by `launch.sh`; the default `change-me-in-production` value must never be used in any networked environment
- `brain/_system/auth.json` and `docker/.env` are in `.gitignore` and must never be committed
